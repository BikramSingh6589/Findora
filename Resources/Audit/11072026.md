## What We Worked On Today

Today's session was focused on fixing broken behaviours in the **Claim flow** and **Conflict Claim flow**, making sure the right things show to the right people at the right time. We also separated the two flows which were previously mixed up, and created documentation for future reference.

---

## Changes Made

---

### 1. Admin Conflict Page — Was Showing Old Resolved Conflicts

**What was wrong:**  
Even after the admin had already resolved a conflict between two claimants, the conflict would still appear in the Admin Conflict Resolution page. Admin had to see already-finished cases cluttering their view.

**What we fixed:**  
The page now only shows conflicts that are genuinely still open and waiting for admin action. Once a conflict is resolved or denied, it disappears from the admin queue automatically.

**How to check:**  
Log in as admin → go to the Conflicts page → resolve a conflict → it should immediately disappear from the list.

---

### 2. Messages / Chat Page — All Chats Were Disappearing

**What was wrong:**  
When a conflict was filed against an item, the original claimant's chat room would vanish completely from their Messages page. They could no longer access any conversation.

**What we fixed:**  
All claim chats are now always visible in the Messages page. Filing a conflict no longer hides anyone's existing chat.

**How to check:**  
Have User A claim an item and start a chat. Have User B file a conflict on the same item. User A's chat should still be visible and accessible in their Messages page.

---

### 3. Real-Time Notifications — Toasts Not Arriving

**What was wrong:**  
When someone sent a message in a chat, the other person was not getting a notification bell or toast popup in real time. The notification system was silently failing to connect properly.

**What we fixed:**  
The notification connection was missing a required security credential (authentication token). We added it so the server can now properly identify who to send the notification to.

**How to check:**  
Have two users open the app. User A sends a message to User B. User B should instantly see a toast notification pop up at the top of the screen, and the bell icon should show a new count.

---

### 4. Community Board — Items Going Missing After Conflict Resolution

**What was wrong:**  
After a conflict was resolved by admin, items were completely disappearing from the Community Board. Users visiting the board would see fewer items than expected.

**What we fixed:**  
Items now always stay visible on the Community Board regardless of their status. Whether an item is claimed, resolved, disputed, or under conflict handover — it stays on the board. The card simply changes its appearance (shows a "CLAIMED" stamp, or "Conflict in Process" button) based on what's happening with it.

**How to check:**  
Post a found item. Have someone claim it. Have someone else conflict it. Admin resolves it. The item card should remain on the Community Board throughout all of this, just with the button changing at each stage.

---

### 5. Conflict Claims — Chat Was Being Opened Between Two Competing Claimants

**What was wrong:**  
When someone filed a conflict claim, the app was accidentally opening a live chat between the conflict filer and the finder of the item. This is wrong — conflicts should be handled by the admin only, with no direct communication between the two competing parties.

**What we fixed:**  
When a conflict claim is filed, the system now permanently blocks the conflict filer from entering a chat room with the finder. Instead, they see an informational screen explaining that the admin is reviewing the case and that they will receive a CFL notification code when it's time to collect.

**How to check:**  
User A claims an item. User B files a conflict against that claim. User B opens their entry in the Messages page — they should see the "Conflict Under Admin Review" screen, NOT a chat input.

---

### 6. Conflict Chat Screen — Was Stuck on "Pending" Even After Admin Resolved

**What was wrong:**  
Even after the admin had made a final decision on a conflict (either in favour of the conflict filer or against them), the conflict filer's screen still showed "Pending Admin Decision" in yellow. It never updated to reflect the actual outcome.

**What we fixed:**  
The screen now reads the actual, real-time status of the conflict from the database and shows the correct outcome:

- If the conflict was resolved in the user's favour → they see a green success screen saying "You Won"
- If the conflict was denied → they see a red screen saying "Claim Denied" with the reason
- If still pending → they see the yellow "Pending Admin Decision" screen

**How to check:**  
File a conflict claim. Admin resolves the conflict in favour of the conflict filer. The conflict filer opens their Messages entry — they should now see a green "Conflict Resolved — You Won!" screen.

---

### 7. Messages Page Badge — Conflict Claims Showing Wrong Status

**What was wrong:**  
In the Messages page list, conflict claims that had been resolved were showing a yellow "Waiting to Scan" badge — which is meant for the QR handover process, not for conflict outcomes. It was confusing and incorrect.

**What we fixed:**  
Conflict claims now have their own correct badge labels in the Messages list:

- Pending → "Admin Review" (yellow)
- Won → "Conflict Won" (green)
- Denied → "Conflict Denied" (red)

**How to check:**  
Go to the Messages page. Any claim that was part of a conflict should show one of the three labels above, not "Waiting to Scan."

### Bug Fixes

**1. Cloudinary Image Uploads Failing (403 / Placeholder Fallback)**

- **What was done:** Modified the `cloudinary.service.ts` configuration logic. Moved the `cloudinary.config()` call from the global scope into a runtime function triggered only when uploads occur.
- **Why it was done:** The service was trying to read `.env` variables before they were fully loaded by `dotenv` during the server boot cycle, which caused all image uploads to fail and fall back to an Unsplash placeholder.
- **How it benefits the project:** Fixes image uploads across the entire application, ensuring user images (found items, lost items, claim proofs) actually save to the Cloudinary account.

**2. Background AI Worker Crashing on Startup (Redis ECONNREFUSED)**

- **What was done:** Disabled the `REDIS_URL` connection parameter in the `.env` file.
- **Why it was done:** The AI Worker was attempting to connect to a local Redis server (which doesn't exist on this environment) causing continuous connection refused errors in the terminal (`ECONNREFUSED 127.0.0.1:6379`).
- **How it benefits the project:** Prevents backend crashes/spam. The application now smoothly uses the built-in "Direct Asynchronous execution" fallback to process AI jobs.

### User Experience

### Bug Fixes

**1. AI Ingestion & Target Skip Pipeline Issue**

- **What was done:** Modified `ai.service.ts` to perform metadata extraction (OCR, keywords, identifiers) inline via a new `processItemAIData` helper. Awaited this inline execution in `triggerMatching` for target items that are not yet analyzed, rather than skipping the comparison loop.
- **Why it was done:** The matching engine was skipping matching passes for un-ingested target items, causing matches to not be generated until a second matching pass occurred.
- **How it benefits the project:** Ensures AI matches are created immediately when items are reported, without skipping comparisons.

**2. Title Case Casing & Duplicate Concept Fields**

- **What was done:** Standardized brand and color matching output strings to Title Case and added strict filtering inside `ai.service.ts` to remove redundant semantic concepts (e.g. `Same Belt` duplicates) and filter out meaningless single-letter tokens (such as `Same N`).
- **Why it was done:** The AI matched fields display was cluttered with duplicate semantic entries and single-letter characters which felt unpolished.
- **How it benefits the project:** Cleans up the AI matching explanation lists inside the item details panel, making them accurate and professional.

**3. Chat Controller Field Mismatches**

- **What was done:** Mapped route-param `itemId` (representing the claim ID) to the `claimId` database field inside `chat.controller.ts` queries and document creation, and removed unsupported schema fields like `type`.
- **Why it was done:** The controller was looking for/creating chat messages using `itemId` but the ChatMessage Mongoose schema uses `claimId`, causing chats to fail to load or save correctly.
- **How it benefits the project:** Fixes the chat history and messaging flow so chats load and persist correctly.

### Features Added

**1. Symmetrical Claim Chats & Finder-Initiated Claims**

- **What was done:** Modified `claim.controller.ts`'s `submitClaim` to allow the finder of an item to initiate a claim/chat room on behalf of the owner of a matched lost item. Also configured the claimant "Open Chat" sidebar action in `ItemDetail.tsx` and "Chat with Owner" button in `AIMatches.tsx` to get/create a claim first to obtain a valid `claimId` before routing.
- **Why it was done:** Chat rooms require a valid `claimId`. Finders initiating chat with matched lost item owners previously routed incorrectly using the item ID.
- **How it benefits the project:** Resolves all chat routing mismatches, enabling finders and claimants to initiate chat rooms seamlessly.

**2. Handover QR Code Generation on Approval**

- **What was done:** Integrated on-the-fly QR code token generation, Cloudinary QR code URL generation, and expiration settings upon admin approval in both `approveClaim` and `mediationResolve` functions of `claim.controller.ts`.
- **Why it was done:** Claims approved by admins lacked a QR code token, causing the Handover QR screen to display empty/blank values.
- **How it benefits the project:** Guarantees that every approved claim has a valid handover QR code immediately ready to display and scan.

**3. Comprehensive Match List & State UI**
- **What was done:** Overhauled `ItemDetail.tsx`'s AI match panel. It now fetches the list of all matches sorted by score and supports: expanding/collapsing detail breakdowns, loading spinner states, 403 restricted access notices, and matching in progress alerts.
- **Why it was done:** The Item Details page previously showed only the first match, and did not handle errors or processing states gracefully.
- **How it benefits the project:** Creates a beautiful, informative, and multi-match UI for participants.

**4. Direct Matching Claims Autopopulation**
- **What was done:** Configured the match list's "Claim This Item" button inside `ItemDetail.tsx` to pass the lost item ID in the navigation state, and updated `ClaimOwnership.tsx` to read it via `useLocation` and pre-populate the target lost item selection automatically.
- **Why it was done:** Users claiming a found item from the matches list had to manually select their lost item again, which was suboptimal for user experience.
- **How it benefits the project:** Simplifies and automates claim submissions when initiating claims directly from matched item screens.

**5. Claim Proof OCR Ingestion & Identifier Verification**
- **What was done:** Configured the `Claim` model to store `ocrData` (OCR text, extracted identifiers, and processed state) and updated `submitClaim` inside `claim.controller.ts` to run OCR text and identifier extraction dynamically on uploaded proof documents. Added a confidence score boost (up to +20 points) if identifiers extracted from the claimant's proof match those of the found/lost items.
- **Why it was done:** Claim proof documents (such as invoices or receipts) were uploaded but not processed via the OCR pipeline, meaning verification confidence was purely answer-based.
- **How it benefits the project:** Connects proof document processing directly to automated claim verification, improving moderator decision quality.

**6. Contact Finder Chat Lock Rules**
- **What was done:** Configured the socket service (`socket.service.ts`) and the frontend chat client (`FinderChat.tsx`) to lock chat rooms and block message transmissions if the claim status is not 'approved' or 'resolved'. Added dedicated Lock banners to notify the user.
- **Why it was done:** Users were previously able to join room sockets and exchange text messages even while their claims were in 'pending' or 'rejected' states.
- **How it benefits the project:** Prevents contact between finders and claimants before official admin/moderator approval, enforcing secure verification.

**7. Real-Time Upload Previews for Claims**
- **What was done:** Configured the claim proof upload grid (`ClaimProof.tsx`) and conflict claim proof box (`ConflictClaim.tsx`) to check file types and render real thumbnail image previews using `URL.createObjectURL(file)` instead of generic document icons/filenames.
- **Why it was done:** Users uploading files on the claim forms had no visual confirmation of the documents they selected.
- **How it benefits the project:** Significantly enhances user confidence by displaying real preview thumbnails of documents/receipts before they hit Cloudinary.

**8. AI Match Error Card Clarification**
- **What was done:** Refactored the matching error conditional checks in `ItemDetail.tsx` to handle 403 authorization restrictions and other backend/network errors separately, showing a lock icon for access restrictions and an error banner with detailed error descriptions for general failures.
- **Why it was done:** General matching errors previously loaded a misleading "AI Match Insights Restricted" box, leaving developers and users unaware of actual underlying network or backend system issues.
- **How it benefits the project:** Differentiates permission issues from system bugs, streamlining error troubleshooting.

**9. Real-Time Upload Size & Type Validation warnings**
- **What was done:** Configured frontend upload event listeners in `ClaimProof.tsx` and `ConflictClaim.tsx` to validate that files are images or PDFs, and are under 10MB limit. If validation fails, it displays dynamic error feedback banners warning the user.
- **Why it was done:** Previously, invalid files or massive files would fail silently during upload or break the backend, leaving users confused.
- **How it benefits the project:** Prevents upload crashes and improves user experience by alerting users of size/type violations before submission.

**10. Moderator Approval Chat Restriction Removal**
- **What was done:** Removed the constraint requiring claim approval before user-to-user chat is allowed. Deleted socket claim status check blocks from `socket.service.ts` and UI-level pending lock banners from `FinderChat.tsx`.
- **Why it was done:** The user requested the removal of the moderator chat lock block to permit direct contact as soon as a claim is initiated.
- **How it benefits the project:** Simplifies the communication flow, allowing finder and claimant to chat immediately.
